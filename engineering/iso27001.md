# ISO 27001

## When to Use
- Client requires ISO 27001 certification
- Building an information security management system (ISMS)
- Enterprise customer requests compliance documentation
- Systematic approach to information security risk management

## Workflow

### Step 1: Scope Definition
Define the ISMS scope:
- What systems are included? (the Next.js application, databases, CI/CD pipeline)
- What data is covered? (customer data, business data, operational data)
- What team members are included?
- What is explicitly out of scope?

### Step 2: Risk Assessment
Identify risks to information security:

| Asset | Threat | Vulnerability | Likelihood | Impact | Risk Level | Treatment |
|-------|--------|--------------|-----------|--------|------------|-----------|
| Customer data | Data breach | Weak access controls | Medium | High | High | Implement RLS + auth |
| Source code | Unauthorized access | No branch protection | Medium | Medium | Medium | Enable branch protection |
| API keys | Key leakage | Keys in code | High | High | Critical | Use environment variables |

### Step 3: Select Controls
ISO 27001 Annex A provides 93 controls across 4 themes:
1. **Organizational** (37 controls): Policies, roles, responsibilities
2. **People** (8 controls): Screening, awareness, training
3. **Physical** (14 controls): Physical security (less relevant for cloud-native)
4. **Technological** (34 controls): Access control, encryption, logging

For a cloud-native Next.js application, focus on:
- A.5.1: Information security policies
- A.8.2: Access rights management
- A.8.5: Secure authentication
- A.8.9: Configuration management
- A.8.24: Use of cryptography
- A.8.28: Secure coding practices
- A.8.32: Change management

### Step 4: Statement of Applicability
Document which controls apply, which do not, and why:
| Control | Applicable | Implementation | Justification |
|---------|-----------|---------------|---------------|
| A.8.5 Secure authentication | Yes | NextAuth + MFA | Protects user accounts |
| A.7.1 Physical security perimeters | N/A | Cloud-hosted on Vercel | No physical infrastructure |

### Step 5: Implement and Document
For each applicable control:
1. Implement the control (technical or procedural)
2. Document the implementation
3. Define evidence of effectiveness
4. Assign an owner

### Step 6: Internal Audit
Before external certification, conduct an internal audit:
- Review each control for effectiveness
- Verify evidence exists
- Identify non-conformities
- Create corrective action plans

## Stack Context
- **Vercel**: Cloud hosting eliminates many physical security controls. Document that infrastructure security is managed by Vercel.
- **Supabase**: Database security, encryption, and access controls.
- **GitHub**: Source code management, change management, and access control.
- **Notion**: ISMS documentation, risk register, and audit findings.
- **NextAuth**: Authentication controls for the application.

## Quality Gate
- [ ] ISMS scope defined
- [ ] Risk assessment completed
- [ ] Statement of Applicability created
- [ ] All applicable controls implemented and documented
- [ ] Internal audit conducted
- [ ] Management review completed

## Anti-Patterns
- **Paper compliance** — documents that describe processes that do not exist.
- **Trying to do everything** — focus on controls relevant to your risk profile, not all 93 controls equally.
- **No risk-based approach** — ISO 27001 is risk-based. Start with risks, then select controls.
- **One-time effort** — ISO 27001 requires continuous improvement. Annual reviews are mandatory.
